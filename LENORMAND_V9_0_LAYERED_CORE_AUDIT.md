# Petit Lenormand v9.0 分層組合核心稽核

更新日期：2026-06-19  
網站版本：v88.7  
引擎方法：`site_petit_lenormand_v9_0_layered_composition`  
公式契約：`site_lenormand_layered_formula_v2`

## 結論

本次撤回 v8.0「只有直接相鄰牌對能形成結論、沒有預製 gate 就拒答」的核心，改成全題型共用的分層組合公式：

> 忠實解析自然語言命題 → 建立題目語義焦點 → 直接牌對形成主要短語 → 連續三張形成次級短句 → 分別彙整支持與風險 → 依淨傾向輸出五級結論 → LLM 只轉寫程式核准主張

此次不是替「廣告／折價活動」新增一次性答案，而是移除「每個新問題都要另建 gate」的架構。一般行動效果與決策適合度現在使用通用命題 `action_effectiveness` 與 `decision_suitability`，其他題型也沿用同一證據管線。

這套公式是本站明示、可測試的現代工程規約，不宣稱是歷史說明書或國際機構認證的唯一雷諾曼公式。

---

## 一、前版為何會變成一問三不知

v8.0 有三個根本問題：

1. **只允許直接相鄰牌對投票**：把三張組合及焦點語境全部降為零，造成大量有效組合在進入模型前被刪除。
2. **每個題型依賴預製 required gates**：自然語言稍微換一種問法就可能得到 `no_formula_for_proposition`。
3. **全有全無門檻**：只要缺少其中一個 gate，就從可能支持直接降為證據不足，無法表達正反並存或較支持／較不支持。

因此「端午節投廣告加折價能否帶起買氣、是否正確」被錯拆成長期副業成功與無公式的一般題，最後只剩狐狸＋鐮刀，並非整個牌陣沒有內容，而是舊引擎先刪除了可讀內容。

## 三、v9.0 共用公式

### 1. 忠實命題解析

解析器先保留使用者真正詢問的對象、行動、結果、判斷與時間範圍，不把短期行銷方案擴張成長期事業成功。

目標案例現在固定拆為：

- `action_effectiveness`：這項行動是否能有效帶動題目中的目標結果；
- `decision_suitability`：這項做法整體是否適合執行。

「端午節」保留為使用者指定的解讀範圍，但在未啟用 timing rules 時，不換算成預測日期。

### 2. 單一語義來源

每個 proposition profile 只定義：

- domain；
- anchors；
- support themes；
- 顯示名稱。

取證池、相關性、支持／風險分數及結論梯度都從同一份 profile 與牌義語義生成，不再同時維護「取證牌表」和「結論 gate 牌表」。這避免兩份規則日後再次不同步。

### 3. 分層證據，而非只有牌對

#### 主要證據：直接相鄰 D

- 權重 3；
- 形成最清楚的局部短語；
- 必須接觸題目焦點，或兩張牌都與該命題高度相關。

#### 次級證據：連續三張 S

- 權重 2；
- 中間牌負責修飾兩側主題；
- 只收錄連續三張，不使用任意長線；
- 不與已選中的同一直接牌對重複灌票；
- 可以進入核准主張，不再被一律降成零權重背景。

#### 不作自動結論的資訊

- 牌號、座標、房屋、鏡像與騎士跳不自行表示時間、因果、意圖或事件；
- hypothetical 人物不因人物牌出現就實體化；
- 年齡、醫療病因及精確時間仍採 fail-closed。

### 4. 一個物理結構只登錄一次

- 每一直接牌對或三張短句只建立一個 evidence ledger entry；
- 同一物理結構不得因不同描述重複計權；
- 相依命題可引用，但不因重複引用增加信心；
- 三張短句若包含已被選為主要證據的直接牌對，會優先避開重複選用。

### 5. 支持與風險分開彙整

系統不再要求每題先滿足若干全有全無 gates，而是分別計算：

- support score；
- risk score；
- balance；
- evidence coverage。

固定負向牌仍維持受控語義：

- 棺材＝結束；
- 鐮刀＝切斷；
- 山＝阻礙；
- 老鼠＝損耗；
- 十字架＝負擔；
- 雲＝不明；
- 鞭子＝衝突／反覆。

負向牌不會被改寫成「先苦後甜」或「必經成長」。

### 6. 五級結論梯度

一般可判命題固定輸出其中一級：

1. 明顯支持；
2. 較支持；
3. 正反並存；
4. 較不支持；
5. 明顯不支持。

只有沒有足夠題目相關證據時才回答證據不足，不再因缺少某個預製 gate 就拒答。

### 7. 建議邊界

牌面可轉譯成與主張直接相符的方向，例如：

- 宜保守測試；
- 支持與代價並存；
- 有推進條件但存在中止或損耗；
- 不宜把局部機會視為穩定結果。

但不能自行生成：

- 廣告應投多少元；
- 折扣應設定幾折；
- 商品、庫存、毛利與客單價數字；
- 具體活動期限或保證成交量。

## 四、提示詞重構

Runtime prompt 現在依序只保留：

1. 最高限制與方法範圍；
2. propositions；
3. 唯一 evidence ledger；
4. 每題程式編譯完成的 claim plan；
5. 實際用到的牌義；
6. 輸出契約。

刪除：

- v8.0 required gate 資料牆；
- `no_formula_for_proposition`；
- 未核准的整副牌資料；
- 任意長線、房屋、鏡像、騎士跳；
- 同一句限制在多個區塊反覆出現；
- 要求模型自行補完缺失門檻的空間。

這與官方提示工程文件的共同方向一致：

- OpenAI：清楚分隔指令與資料，建立可重複的 evals，不依賴只看少數範例的人工感覺；
- Anthropic：用一致、具描述性的 XML 標籤區分指令、背景、輸入與輸出；
- Google Cloud：清楚定義任務並拆解複雜流程，以提高可控性、除錯性與準確度。

官方來源：

- https://developers.openai.com/api/docs/guides/prompt-engineering
- https://developers.openai.com/api/docs/guides/evals
- https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/multishot-prompting
- https://docs.cloud.google.com/gemini-enterprise-agent-platform/models/prompts/prompt-design-strategies
- https://docs.cloud.google.com/gemini-enterprise-agent-platform/models/prompts/break-down-prompts

## 五、目標案例驗證

題目：

> 這次我端午節，我副業蝦皮下一千元廣告並且辦折價活動，是否能順利帶起買氣？是否正確？

新版解析結果：

- `action_effectiveness`
- `decision_suitability`

在固定測試牌序下，兩題都能形成實質五級結論，不再輸出「本站沒有公式」：

- 行動效果：`正反並存`；
- 決策適合度：`正反並存`。

實際結論仍會隨抽出的牌陣改變，但永遠由相同分層公式產生，不需為這一句問題新增專屬 gate。

## 八、誠實限制

本次根治的是會造成「一題一修」的核心架構：

- 題型專屬 hard gates；
- 只有直接相鄰才能回答；
- 無預製公式就拒答；
- 支持與風險只能全有全無；
- 取證表與判斷表雙重維護。

它不代表任何占卜結果具有科學預測效力，也不能保證所有未來自然語言永遠零解析錯誤。新版的可驗證目標是：新問題可由同一通用公式產生可回答、可追溯、有限度且不重複灌票的解讀，而不是每次新增一題就再寫一套規則。
