# Petit Lenormand v10.0 四牌陣統一編譯器稽核

日期：2026-06-19  
交付版本：網站 v88.8／雷諾曼引擎 v10.0

## 一、結論

本次不是再為單一問題增加例外條件，而是修正共用資料流：

> 問題解析 → 牌陣幾何 → D牌對／S三張短句 → 命題相關性 → 正反整合 → claim_plan → claim_evidence → AI轉寫

三張、五張、九張與36張現在全部先完成同一套 `claim_plan`／`claim_evidence` 編譯；不再發生小牌陣只有抽牌資料、卻缺少核准主張而無法解讀的斷層。

本引擎可提高的是**規則一致性、證據可追溯性、題意忠實度與輸出穩定性**。雷諾曼屬占卜系統，沒有科學方法能保證未來事件預測「一定準確」，因此本版本不宣稱客觀預測正確率。

## 三、四種牌陣共用核心與各自深度

### 1. 三張線

幾何：

- 2個直接相鄰牌對；
- 1個完整三張短句；
- 左／中／右採條件、核心、影響的中性位置，不在未啟用方向規則時硬寫成過去、現在、未來。

輸出深度：2至3段。  
適用：單一趨勢、近期狀態、是否順利、注意事項。

### 2. 五張線

幾何：

- 4個直接相鄰牌對；
- 3個滑動三張短句；
- 第3張為中心；
- 額外保留近身三張及1↔5、2↔4兩組對稱脈絡，但不重複增加信心。

輸出深度：3至5段。  
適用：核心、左右修飾、支持、阻力與可行方向。

### 3. 九宮格

幾何：

- 第5張為中心；
- 水平、垂直及對角的合法相鄰關係共20組；
- 3橫列、3直欄、2對角線，共8條完整三張線；
- 中心十字作主要局部脈絡。

輸出深度：3至6段。  
適用：多面向矛盾、人物關聯、局勢核心、支持與風險交叉分析。

### 4. 36張大牌陣

幾何：

- 維持4×8＋4；
- 使用人物牌與命題主題牌作焦點；
- D讀焦點附近的局部語句；
- S讀連續三張短句；
- 最多保留3個主要焦點鄰域，避免未核准資訊形成資料牆；
- 房屋與全局框架只補充已成立主張，不單獨升級 certainty cap。

輸出深度：4至7段。  
適用：多命題、整體局勢、人物附近、主題牌群與正反矛盾。

## 四、此次根治的核心問題

### 1. 小牌陣缺少 claim_plan

舊版三張、五張與九張可能只輸出 `<drawn_cards>`，最高規則又要求只能轉寫 `claim_plan`，造成模型只能回答「無法解讀」。

新版：所有非比較牌陣一律進入相同編譯器，自動建立：

- propositions；
- evidence ledger；
- evidence packets；
- claim plan；
- claim evidence；
- analysis requirements；
- approved dictionary。

### 2. 一題一套 gate

舊版必須預製題型公式，沒有 gate 就拒答。

新版改為通用語義軸，例如：

- `outcome_tendency`；
- `risk_guidance`；
- `action_effectiveness`；
- `decision_suitability`；
- `relationship_intent`；
- `relationship_longevity`；
- `business_success`；
- `life_guidance`。

未命中特殊高風險限制時，可回落到通用正反整合，不再因缺少逐題 gate 而拒答。

### 3. D與S互相排斥

舊版曾把S降成完全不投票，也曾因S包含D而直接刪掉S，兩種做法都會損失組合語義。

新版：

- D是主要局部語句；
- S是較低權重完整短句；
- 兩者可同時保留；
- 若共享邊，S採0.7或0.45的語義新穎度折減；
- 同一物理結構在全題帳本只登錄一次。

### 4. 深度靠資料量而非證據

新版只將 claim evidence 核准的結構交給AI形成主張；spread context只能補充已成立主張的樣貌。深度來自：

- 核心判斷；
- 有利結構；
- 阻力；
- 正反矛盾；
- 可能表現；
- 未知邊界。

不再靠重複列出房屋、鏡像、騎士跳或所有鄰牌製造篇幅。

## 五、提示詞工程依據

本次提示詞重排為：

1. 結果與最高限制；
2. 結構化命題；
3. 核准證據帳本；
4. 每項主張的 evidence reference；
5. 牌陣脈絡；
6. 輸出契約。

採用的官方工程原則：

- OpenAI指出較短、結果優先的提示通常優於過度規定流程的舊提示堆疊，並建議清楚提供目標、限制、證據與輸出要求：<https://developers.openai.com/api/docs/guides/prompt-guidance>
- OpenAI模型最佳化流程要求先建立eval基線，再用代表性測試反覆驗證：<https://developers.openai.com/api/docs/guides/model-optimization>
- Google官方提示設計文件強調明確目標、內容與結構，並以系統化測試優化：<https://docs.cloud.google.com/gemini-enterprise-agent-platform/models/prompts/prompt-design-strategies>

本次最大壓力測試提示詞為11,985字元，低於自動測試設定的12,000字元上限。
