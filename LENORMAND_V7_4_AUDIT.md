# Petit Lenormand v7.4 稽核與根治紀錄

- 專案版本：v88.4 / Lenormand v5.4
- 稽核日期：2026-06-19
- 基線：v88.3 有變動檔案覆蓋至 `mytool-blue-master.zip`
- 目標案例：`我對人生有些迷茫 請給我建議`
- 修正原則：改資料契約與取證流程，不以新增單題禁詞或輸出後修補處理

## 一、結論

v7.3 已經有 `claim_evidence`，但在「人生迷茫／請給建議」這類開放式問題上仍未真正形成證據約束。問題不只是提示詞偏長，而是上游資料契約有三個根因：

1. `general` 類型的焦點與支援牌範圍過廣，會把感情、工作、財務、穩定、成功等不同領域一起送入同一題。
2. 唯一的 `A1` 是「依核心證據與反證給出結論」，沒有可驗證的語意邊界；它等同允許模型自行組裝任何合理建議。
3. 舊分簇會將同一核心牌周圍的正向、負向與題外牌打包成同一 C。即使 prompt 只准使用某個 C，仍可能把該 C 內未真正支援主張的牌一併帶入。

因此，v7.3 的「證據血緣」在此類題目上只是形式存在，不能充分阻止題外延伸。

## 二、原 AI 解讀的具體問題

### 2.1 可由牌面支持的部分

以下內容在受控字典與牌對範圍內，方向基本合理：

- 紳士與鑰匙可描述「問卜者與解答／重要判斷有直接連結」。
- 鑰匙、老鼠、鞭子可描述「解答或判斷受到損耗與反覆干擾」。
- 心、幸運草、太陽可描述「熱情與小機會／正向支點同時存在」。
- 太陽、錨、房屋可描述「成功、穩定與根基的象徵結構」。
- 心與棺材／鐮刀／十字架可描述「熱情領域同時出現結束、切斷與負擔」。

### 2.2 越過核准證據的部分

原解讀加入了以下沒有被 v7.3 `approved_claims` 明確批准的操作或心理處方：

- 「把注意力收回自己」：這是一般心理建議，不是受控牌義可直接推出的主張。
- 「停止讓所有選項同時占據注意力」：主要來自 selected context 中的十字路口，但 selected context 按契約只能描述已成立主張，不能獨立產生建議。
- 「辨認哪些執著只剩負擔」：把負擔牌擴寫成特定心理執著，超過字典與 claim plan。
- 「哪些選擇能持續、能落地，也不會讓你長期失去力量」：把穩定結構延伸成對未來結果的條件式保證。

這些句子未必是生活建議上的壞話，但在本站的「證據優先」契約下屬於未批准推論。

### 2.3 深度看似足夠，實際證據獨立性不足

原輸出第一段一次列出七個 D，但其中多數來自同一局部 C。雖然規則寫了「同一 C 只算一次」，模型仍看到大量牌對，容易將同一局部現象誤當成多筆獨立支持。這是輸入上下文設計問題，不能只靠一句禁止規則修正。

### 2.4 水晶推薦基準仍不穩定

原 prompt 的唯一主張 A1 太寬，導致「正文最主要 approved claim」沒有可機械辨認的具體內容。黑曜石雖能對應負向牌，但系統無法證明「限制／界線」一定是本題唯一主要方向。根因仍是 claim plan 不具語意結構，而不是候選清單本身。

## 三、史料與「現代正統」核實

### 3.1 可直接核實的歷史事實

英國博物館館藏頁明載《Das Spiel der Hofnung》為完整 36 張牌，並含一份 4 頁印刷說明。這可支持本站將 36 張牌組與 4 頁說明列為可核實史料。

官方館藏頁：
- https://www.britishmuseum.org/collection/object/P_1896-0501-495

### 3.2 不能宣稱為「唯一現代正統」的部分

官方館藏沒有替本站的 D／S／C、房屋、鏡像、騎士跳、門檻、計分或 certainty cap 背書。這些是網站為了可驗證、一致性與避免越權而建立的工程規約。

因此正確說法是：

- 歷史層：36 張牌與保存說明書可核實。
- 工作詞典層：本站採用受控的現代牌義集合。
- 工程層：D／S／C、分簇、血緣、門檻與輸出限制是本站方法。

不存在一份由博物館或國際標準組織發布、能證明本站所有現代技法為「唯一正統」的官方標準。v7.4 保留上述分層，不再將工程規約冒充史料。

## 四、提示工程技術核實

### 4.1 OpenAI 官方文件

OpenAI 官方 Prompt Engineering 指南建議：

- 以不同權限層級放置高階 instructions；
- 使用 Markdown 與 XML 劃分清楚的邏輯邊界；
- 複雜正式系統應固定模型快照並建立 evals；
- 只加入真正相關的 context。

來源：
- https://platform.openai.com/docs/guides/prompt-engineering

v7.4 對應修正：最高規則前置；把證據、主張、限制與輸出契約分成固定標籤；新增可重現的行為測試。

### 4.2 Anthropic 官方文件

Anthropic 官方文件指出，混合指令、背景、範例與變數時，使用一致且具描述性的 XML 標籤可降低誤解；範例應貼近真實案例並涵蓋邊界情況。官方也明確指出，不是所有失敗都應靠 prompt engineering 處理。

來源：
- https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview
- https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/multishot-prompting

v7.4 對應修正：沒有試圖再加一長串禁詞壓模型，而是修正命題類型、分簇、核准證據視圖與 claim plan。

### 4.3 Google Cloud 官方文件

Google 官方 Prompt Health Checklist 明列下列常見失敗：

- 冗長、含糊或不必要的措辭；
- 重複指令與範例；
- 無關指令與範例；
- 太多不同任務塞在單一提示；
- 彼此衝突的內部參照；
- 缺少明確輸出格式；
- 未針對缺失資料建立處理方式。

來源：
- https://docs.cloud.google.com/vertex-ai/generative-ai/docs/learn/prompts/prompt-design-strategies

v7.4 對應修正：人生建議題不再載入財務案例、原始鄰近清單、鏡像、騎士跳、交會與未核准 C；提示詞由約 12K 級上下文縮至約 5.3K 字元。

## 五、v7.4 根治內容

### 5.1 新增 `life_guidance` 命題

下列語意在沒有其他明確領域時，進入獨立命題：

- 人生迷茫／茫然
- 找不到方向／未來方向
- 請給我建議
- 下一步怎麼走

這類題目不再由 `general` 同時召回感情、財務、商業與成功結構。

### 5.2 將通用 A1 拆為四類可驗證主張

人生建議題改為：

1. 方向是否有可釐清的線索；
2. 實際出現哪些受控負向干擾；
3. 牌面能批准哪一種象徵性篩選原則；
4. 牌面不能指定什麼。

每項主張最多連結兩個 C；不同主張不得重複使用同一 C。

### 5.3 取證階段先做相關性過濾

人生建議題不再因「問卜者牌附近」就保留所有鄰牌。只有：

- 本題焦點牌；或
- 受控的強負向牌

才可進入直接牌對候選。這會在模型看到資料前先排除題外人物、財務與公開場域牌。

### 5.4 分簇依證據角色與極性拆開

人生建議題的分簇不再只依共同核心牌。正向支點、負向干擾與中性方向訊號分開，避免同一 C 同時含「解答、女性、財務、負擔、社交」等互不相干內容。

### 5.5 建立 `approved_evidence_scope`

Runtime prompt 不再輸出完整 `evidence_catalog` 與所有 `core_clusters`。它先由 `claim_evidence` 建立核准視圖，只輸出：

- 被至少一項 approved claim 引用的 C；
- 這些 C 真正包含的 D／S；
- 這些核准 C 對應的主題摘要。

未被主張使用的結構不進入模型上下文。

### 5.6 selected context 降為核准主題摘要

人生建議題不再輸出：

- N-P / N-T 鄰近清單；
- 鏡像；
- 騎士跳；
- 線段交會；
- 四角與中心框架。

只保留核准 C 已有的主題摘要，且 certainty effect 固定為 none。

### 5.7 analysis requirements 改讀核准證據

段落數、正向、風險、反覆與矛盾需求，都由核准 evidence view 計算。未被 claim plan 採用的負向牌，不再強迫 AI 補寫風險或矛盾。

### 5.8 水晶推薦只讀核准證據

人生方向題可以召回紫水晶作候選，但：

- 候選不是推薦順位；
- 最終仍必須對應正文主要 approved claim；
- 無吻合者必須輸出不強行推薦；
- 礦物事實與象徵說法分開。

紫水晶的礦物事實使用 GIA：石英、SiO₂、莫氏硬度 7。

來源：
- https://www.gia.edu/amethyst

黑曜石的礦物事實使用 USGS：天然火山玻璃，通常由高矽、黏稠岩漿快速冷卻形成。

來源：
- https://www.usgs.gov/observatories/yvo/news/yellowstones-tool-making-lava-flows
